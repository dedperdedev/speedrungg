# ClickUp MCP Workflow

How agents (especially `project-manager`) integrate with ClickUp for backlog and sprint management. Used when the user keeps tasks in ClickUp and wants Claude to read/write tasks instead of (or in addition to) `production/backlog.md`.

## When to use

- User stores tasks in ClickUp (not just markdown)
- Multi-person workflow where ClickUp is single source of truth
- Sprint planning needs cross-task visibility (assignee, due dates, time estimates)
- Time tracking is required

## When NOT to use

- Solo dev with markdown-only workflow — overhead not worth it
- Sensitive game design notes — ClickUp is cloud; sensitive docs belong in Obsidian/local
- Throwaway prototype — don't pollute your real project workspace

## Setup

1. **Get ClickUp API token**: ClickUp Settings → Apps → API Token (personal, not OAuth — simpler for solo)
2. **Get team ID**: Find in ClickUp URL `https://app.clickup.com/<TEAM_ID>/...`
3. **Get target list ID**: Where new tasks go by default
4. **Add to `.env`** (gitignored):
   ```
   CLICKUP_API_TOKEN=pk_xxxxx
   CLICKUP_TEAM_ID=12345
   CLICKUP_DEFAULT_LIST_ID=678901
   ```
5. **Enable in `.claude/mcp.json`**: change `"enabled": false` → `true`
6. **Verify**: in Claude Code, run `/mcp` — should show `clickup` connected

## What agents can do (via MCP tools)

Standard ClickUp MCP servers expose tools like:
- `get_lists` — find available lists in a team
- `create_task` — make a new task with title, description, status, assignee
- `update_task` — modify status, due date, assignee
- `get_task` — read task details
- `list_tasks` — query tasks (filtered by list, status, assignee, etc.)
- `add_comment` — append to task discussion
- `set_status` — move task between Open/In Progress/Review/Closed
- `track_time` — log time spent (if your plan supports time tracking)

## Agent integration patterns

### `project-manager` workflow

Each `/sprint-plan`:
1. Fetches open backlog from ClickUp via `list_tasks` (filter: status=Open, sort by priority)
2. Cross-references with `production/backlog.md` to detect drift
3. Presents calibrated sprint plan to user
4. After approval: creates ClickUp tasks for new items, updates statuses for moved items
5. Mirrors final sprint to `production/sprint/current.md` for offline access

### `release-manager` workflow

During `/release-checklist`:
1. Pulls all tasks tagged "release-blocker" from ClickUp
2. Checks each against the checklist (security audit done? perf budget passed? etc.)
3. Updates task statuses as items resolve

### Bidirectional sync discipline

ClickUp ↔ markdown sync risks divergence. Two patterns:

**Pattern A — ClickUp as truth (recommended for teams):**
- All task creation via ClickUp MCP
- Markdown files are read-only mirrors generated by `/sprint-plan` and `/sprint-retro`
- Manual edits to markdown are advisory; ClickUp wins on conflict

**Pattern B — Markdown as truth (recommended for solo):**
- Tasks created/edited in markdown
- ClickUp sync only at sprint boundaries
- Use ClickUp for visualization (kanban view, calendar) but not source

Pick one explicitly. Document the choice in `production/backlog.md` header.

## Calibration on what ClickUp data is worth

ClickUp is good at: state (status), ownership (assignee), time (due date, time spent).
ClickUp is bad at: design rationale, cross-task relationships, calibrated decisions.

Rich content (design rationale, [HYPOTHESIS] labels, evidence chains) belongs in markdown that the task description *links* to. Don't try to put a 500-word design spec in a ClickUp task description.

## Anti-patterns

- **Putting calibration labels inside ClickUp task descriptions** — they get lost; ClickUp's text formatting is shallow
- **Letting Claude auto-create dozens of tasks at once** — review before commit; deletion is friction
- **Multi-person dependency without explicit sync ritual** — divergence will happen
- **Leaving stale tasks open for months** — kills signal-to-noise; archive aggressively

## Operational checklist

- [ ] Token in `.env`, not committed
- [ ] Team ID + default list configured
- [ ] One sync pattern chosen and documented
- [ ] `/sprint-plan` uses ClickUp data
- [ ] `/sprint-retro` writes back to ClickUp
- [ ] Closed tasks archived monthly
