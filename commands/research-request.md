# /research-request

The official way for creative-director / product-owner / project-manager to ask the Research Hub for something specific. Specialists in the main studio CANNOT use this — go through your Tier 1 first.

## Protocol

1. Leadership fills the brief:

```
Question: <specific, decision-driving>
Decision it informs: <what choice this answers>
Deadline: <when needed>
Max effort: <S / M / L>
Format: <quick note / structured report / digest item>
```

2. `research-director` reviews:
   - Is the question well-formed (not "research market for me" — too vague)?
   - Which hub agent(s) own this?
   - What's the rough plan?
   - Confirm scope with leadership before starting.

3. Hub executes with hard guardrails (citations, labels, dates, confidence).

4. Returns answer in agreed format. Logs in `docs/research/requests/<date>-<slug>.md`.

## Effort sizes

| Size | Effort | Examples |
| --- | --- | --- |
| S | Hours | Quick fact-check, single-source check |
| M | 1-3 days | One-source deep dive, comp profile refresh |
| L | Up to 2 weeks | Audience deep-dive, multi-source synthesis |
| XL | Refused — break into smaller requests |

## Anti-patterns

- Vague questions ("understand the market")
- No decision attached (research with no consumer wastes everyone's time)
- Bypassing leadership filter (specialists ad-hoc DM-ing researchers)
- Multiple unrelated questions in one request
- Deadline of "tomorrow" for L-size work — push back, don't over-promise
